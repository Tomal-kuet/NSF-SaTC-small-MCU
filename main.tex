\section{Motivation and Research Challenges}
% ==========================================================
\subsection{Why embedded trust anchors need to change}
Existing embedded TEEs are useful because they isolate security-sensitive code from a rich execution environment (REE), but they do not fully solve the trust problem for modern embedded systems. First, their hardware trusted computing base (TCB) is typically fixed at design time. A trusted domain may include unnecessary peripherals, broad memory privileges, and privileged drivers that violate least privilege. Second, many TEEs are proprietary or architecture-specific, limiting independent inspection and customization. Third, a TEE is often treated as a single trusted box. This ignores the fact that TEE software is commonly written in C/C++ and may contain bugs. Fourth, runtime attestation often proves only that the expected binary was loaded, not that the firmware followed the expected behavior. Finally, modern MCUs are no longer uniformly simple. Caches, branch prediction, fast secure/non-secure transitions, and hardware cryptographic features improve performance but may change the microarchitectural threat model.

\subsection{Preliminary foundations from the dissertation}
\textbf{FPGA-based customizable TEEs.} \byotee{} explores the use of SoC FPGAs as configurable trust anchors. The key insight is that an FPGA can instantiate application-specific enclaves with dedicated softcore CPUs, BRAM-based memory, peripherals, and interconnects. Because co-resident enclaves need not share processors or caches, the design can reduce several attack surfaces associated with monolithic TEEs. Figure~\ref{fig:byotee} shows the \byotee{} architecture and workflow, where a development-time toolchain generates protected FPGA images and security-sensitive applications (SSAs), and runtime enclaves execute isolated workloads.

\textbf{Hardware-assisted control-flow attestation.} \enola{} addresses the challenge of attesting embedded firmware execution without transmitting unbounded traces. It uses a bounded trace representation and hardware-assisted pointer-authentication measurements to create compact evidence of runtime behavior. Figure~\ref{fig:enola} shows the \enola{} workflow, where compiler instrumentation, secure measurement, and verifier-side reconstruction are combined to attest execution paths.

\subsection{Scientific questions}
The proposed work is driven by four research questions. \textbf{RQ1:} How can embedded devices construct minimal and attestable trusted execution environments after deployment? \textbf{RQ2:} How can a verifier check not only which code was loaded, but also how the code executed and what I/O path it used? \textbf{RQ3:} How can a trusted domain be internally compartmentalized so that one vulnerable driver or library cannot compromise the full TEE? \textbf{RQ4:} How should embedded trust anchors be designed if modern MCUs support transient-execution leakage primitives?

\section{System and Threat Model}
% ==========================================================
\textbf{System model.} We consider embedded systems that contain a microcontroller or embedded processor, optional FPGA fabric, memory protection hardware, secure boot support, hardware cryptographic primitives, and peripheral interfaces. The protected workloads include bare-metal firmware, RTOS tasks, trusted services, sensor-processing applications, and security-sensitive accelerators. The SoC FPGA model includes a hard processor system, an FPGA configuration module, secure key storage, FPGA fabric, BRAM, interconnects, and peripheral I/O. Figure~\ref{fig:socfpga} illustrates the relevant architecture.

\begin{figure}[t]
	\centering
	\includegraphics[width=0.88\linewidth]{figs/fig03_soc_fpga_architecture.png}
	\caption{SoC FPGA architecture used as the hardware basis for reconfigurable trust anchors. The FPGA configuration module and FPGA-side secure storage form the root of trust for loading protected bitstreams.}
	\label{fig:socfpga}
\end{figure}

\textbf{Adversary model.} The adversary may compromise the normal-world software stack, including applications, RTOS components, drivers, hypervisors, or host operating systems. The adversary may exploit memory corruption bugs, malicious peripheral inputs, confused-deputy interfaces, or vulnerabilities in trusted components. The adversary may attempt control-flow hijacking, data tampering, I/O spoofing, compartment escape, or microarchitectural leakage. We assume standard cryptographic primitives are secure and that device roots of trust are provisioned correctly. Physical attacks and invasive hardware modification are outside the primary scope, though the project will consider realistic deployment constraints such as limited memory, limited FPGA resources, and timing-sensitive workloads.

\textbf{Security goals.} The system should provide confidentiality and integrity for trusted workloads, isolation among enclaves and compartments, attestation of hardware and software configuration, attestation of runtime behavior, and practical protection against transient leakage at trusted boundaries.

\section{Thrust 1: Reconfigurable and Attestable Embedded TEEs}
% ==========================================================
\subsection{Problem statement}
A fixed TEE gives every trusted application the same hardware privilege boundary. This is misaligned with embedded workloads, where different applications require different peripherals, timing constraints, memory sizes, and threat models. This thrust asks: \emph{Can we synthesize and attest minimal hardware/software trust anchors from application-specific requirements?}

\subsection{Task 1.1: Policy-guided enclave synthesis}
We will design a policy language for specifying enclave requirements, including CPU type, memory size, peripheral access, accelerator modules, firmware libraries, communication buffers, and attestation requirements. A synthesis tool will translate these policies into FPGA enclave designs. The tool will minimize the hardware/software TCB by including only required resources. For example, an audio-processing SSA should not inherit access to a camera peripheral, and a cryptographic service should not inherit display or network drivers.

\subsection{Task 1.2: Hardware/software configuration attestation}
We will design attestation mechanisms that cover the FPGA bitstream, enclave topology, firmware image, loaded application, memory map, peripheral bindings, and host-enclave communication channels. The attestation report will bind the evidence to a fresh nonce from the verifier and to a device key rooted in secure configuration. The report will allow the verifier to check that the configured enclave matches the expected application policy.

\subsection{Task 1.3: Safe enclave reconfiguration}
Many embedded deployments need updates after manufacturing. We will investigate safe reconfiguration mechanisms that allow enclaves to be updated, replaced, or specialized without violating isolation. The key challenge is to prevent stale trust assumptions: a verifier should not accept a runtime report generated under a previous enclave topology. We will bind reconfiguration events to attestation epochs and design rollback-resistant metadata.

\subsection{Technical challenges and approach}
The main challenge is maintaining a small TCB while supporting general-purpose software. We will combine FPGA-level isolation, BRAM-based memory, narrow shared-memory channels, and hardware-enforced peripheral bindings. We will also develop static checks to detect unsafe interconnects, untrusted debug modules, DMA paths, and hidden peripheral routes. The toolchain will emit a machine-readable enclave manifest used by both the hardware generator and the verifier.

\subsection{Expected outcomes}
This thrust will produce a policy-driven enclave synthesis tool, a hardware/software manifest format, and a configuration attestation protocol for reconfigurable embedded trust anchors.

\section{Thrust 2: Runtime Behavior Attestation}
% ==========================================================
\subsection{Problem statement}
Configuration attestation proves that a trusted environment and binary were loaded, but it does not prove that the binary followed an intended path or used the intended I/O. This thrust asks: \emph{Can embedded systems provide compact, low-overhead runtime evidence of control-flow, data-flow, and I/O behavior?}

\subsection{Task 2.1: Control-flow attestation for realistic embedded execution}
We will extend \enola{} to support long-running firmware, interrupts, callbacks, and RTOS tasks. A major challenge is that embedded systems are event-driven; interrupt service routines and scheduler events introduce control-flow transfers that are not captured by a simple application CFG. We will build an interrupt-aware trace model that distinguishes normal branches, function returns, exception entries, exception returns, context switches, and trusted-compartment transitions.

\subsection{Task 2.2: Hardware-assisted measurement and verification}
We will use hardware cryptographic primitives, such as pointer authentication or lightweight MAC engines, to reduce measurement overhead. The measurement scheme will generate compact evidence whose size is bounded by program structure rather than raw runtime length. The verifier will use static CFG information, attestation manifests, and runtime evidence to reconstruct or validate allowed execution behavior.

\subsection{Task 2.3: Data-flow and I/O-flow evidence}
Control-flow evidence alone may be insufficient: an execution can follow a legal path but operate on maliciously substituted data. We will extend attestation to cover security-critical data dependencies and peripheral interactions. Examples include sensor input hashes, actuator command commitments, DMA buffer identities, and enclave-to-peripheral routing evidence. The output report will bind the software path to the I/O path.

\subsection{Expected outcomes}
This thrust will produce algorithms and prototypes for control-flow, data-flow, and I/O-flow attestation that are suitable for constrained embedded devices.

\section{Thrust 3: Compartmentalized Trust Anchors}
% ==========================================================
\subsection{Problem statement}
A monolithic trusted domain remains fragile: a single bug in a parser, driver, crypto wrapper, or application can compromise all trusted assets. This thrust asks: \emph{Can embedded trust anchors enforce least privilege internally and attest that compartment boundaries were respected?}

\begin{figure}[t]
	\centering
	\includegraphics[width=0.95\linewidth]{figs/fig05_compartmentalized_trust_anchor.png}
	\caption{Proposed compartmentalized trust anchor. Trusted services are decomposed into compartments with explicit memory, key, and peripheral privileges. A compartment monitor enforces transitions and provides compartment-aware evidence to the verifier.}
	\label{fig:compartment}
\end{figure}

\subsection{Task 3.1: Decomposing trusted software}
We will decompose trusted software into compartments such as attestation logic, cryptographic services, peripheral drivers, parsers, application logic, and communication buffers. Each compartment will have a minimal set of privileges. For example, a camera driver may access the camera peripheral and input buffer but not the attestation key, while the attestation compartment may access measurement state but not raw sensor streams unless explicitly authorized.

\subsection{Task 3.2: Hardware-assisted compartment isolation}
We will investigate several enforcement mechanisms: MPU-based isolation on MCUs, TrustZone-M state separation, FPGA memory firewalls, compiler-based software fault isolation, and hybrid mechanisms. The design will optimize for the constraints of embedded systems, including small memory, limited MPU regions, interrupt latency, and deterministic timing.

\subsection{Task 3.3: Compartment-aware attestation}
We will extend runtime reports to include compartment identity, compartment transitions, memory access policies, peripheral access evidence, and violations. This creates a stronger claim than conventional TEE attestation: the verifier can check not only that the trusted software ran, but also that its internal privilege boundaries were enforced during execution.

\subsection{Expected outcomes}
This thrust will produce a compartmentalization framework for embedded trust anchors and a verifier protocol for checking compartment-level behavior.

\section{Thrust 4: Transient-Resilient Embedded Isolation}
% ==========================================================
\subsection{Problem statement}
Many security designs for MCUs assume simple in-order execution and limited microarchitectural state. Newer MCUs, however, increasingly include caches, branch prediction, prefetching, and other performance features. This thrust asks: \emph{Do emerging MCUs support transient-execution leakage primitives, and how should embedded trust anchors defend against them?}

\begin{figure}[t]
	\centering
	\includegraphics[width=0.95\linewidth]{figs/fig06_mcu_transient_attack_measurement_loop.png}
	\caption{Proposed transient-attack measurement loop for MCUs. The framework tests cache timing, speculative cache loading, branch predictor behavior, transient windows, and secret-dependent cache footprints under controlled configurations.}
	\label{fig:transient}
\end{figure}

\subsection{Task 4.1: Measuring transient primitives on MCUs}
We will build a measurement framework for cache timing, branch prediction, speculative cache loading, transient execution windows, and secret-dependent cache footprints. The framework will configure relevant architectural features, trigger candidate transient behaviors, measure timing through cycle counters, and evaluate whether secrets can influence observable microarchitectural state.

\subsection{Task 4.2: Leakage analysis of embedded TEEs and attestation}
We will analyze trusted boundaries, including TrustZone-M transitions, FPGA enclave interfaces, attestation measurement routines, compartment switches, and peripheral handlers. The goal is to determine whether secrets from trusted compartments can influence shared caches, predictors, or timing-visible memory state.

\subsection{Task 4.3: Defense mechanisms}
We will design defenses such as cache partitioning, cache flushing, speculation barriers, constant-time enclave interfaces, compiler instrumentation, enclave scheduling rules, and FPGA-side physical isolation. We will compare defenses in terms of security coverage, runtime overhead, code size, and deployability.

\subsection{Expected outcomes}
This thrust will produce a systematic methodology for evaluating transient leakage on modern MCUs and a set of defenses for transient-resilient embedded trust anchors.

\section{Integration: End-to-End Verifiable Embedded Trust}
% ==========================================================
The four thrusts will be integrated into a single attestation architecture. A verifier will send a fresh challenge to the device. The device will respond with a composite report:
\[
\mathsf{Report}=\langle H_{bitstream}, H_{enclave}, H_{firmware}, H_{app}, IO_{binding}, C_{comp}, T_{runtime}, D_{critical}, nonce\rangle_{\sigma},
\]
where $H_{bitstream}$ captures the FPGA configuration, $H_{enclave}$ captures the enclave topology, $H_{firmware}$ and $H_{app}$ capture loaded software, $IO_{binding}$ captures peripheral routes, $C_{comp}$ captures compartment policy and transitions, $T_{runtime}$ captures control-flow evidence, $D_{critical}$ captures selected data-flow and I/O-flow evidence, and $nonce$ prevents replay. The verifier checks this evidence against an application policy.

The integrated system will allow stronger claims than current embedded attestation mechanisms. Rather than merely proving that a binary was loaded, it will prove that a minimal trusted execution substrate was configured, the expected software ran, trusted components remained compartmentalized, and the runtime behavior was consistent with the expected control-flow and I/O policy.

\section{Evaluation Plan}
% ==========================================================
We will evaluate \sys{} along five dimensions.

\textbf{Security evaluation.} We will test resistance to control-flow hijacking, return-oriented programming, malicious peripheral input, data tampering, I/O spoofing, compartment escape, and transient leakage. We will use synthetic attacks, real firmware vulnerabilities when available, and microbenchmarks for cache/speculation behavior.

\textbf{Performance evaluation.} We will measure runtime overhead, attestation latency, interrupt latency, context-switch overhead, compartment-switch overhead, and reconfiguration cost. Measurements will be taken on representative Cortex-M boards and SoC FPGA platforms.

\textbf{Memory and hardware cost.} We will report code size, RAM usage, stack usage, FPGA LUT/FF/BRAM/DSP utilization, and power/energy overhead. For FPGA enclaves, we will measure the cost of softcore CPUs, memory, interconnects, peripheral wrappers, and hardware attestation modules.

\textbf{Scalability evaluation.} We will evaluate increasingly complex bare-metal applications, RTOS workloads, multi-compartment trusted services, and multi-enclave FPGA configurations. We will compare evidence size and verification time against trace-based and hash-chain-based baselines.

\textbf{Deployability evaluation.} We will demonstrate case studies in sensor processing, secure audio or camera pipelines, cryptographic services, and embedded control logic. The goal is to evaluate whether the mechanisms can be deployed under realistic resource and timing constraints.

\section{Preliminary Results and Feasibility}
% ==========================================================
The feasibility of the project is supported by the dissertation prototypes. \byotee{} implemented an FPGA-based trusted execution infrastructure with hardware generation, firmware, SSA packaging, and attestation mechanisms. It demonstrated the construction of multiple isolated enclaves and evaluated hardware resource utilization, software TCB size, and execution performance. \enola{} implemented compiler instrumentation and runtime support for efficient control-flow attestation on Cortex-M systems. Figure~\ref{fig:pass} shows the compiler pass pipeline used in the preliminary system.

\begin{figure}[t]
	\centering
	\includegraphics[width=0.90\linewidth]{figs/fig08_enola_compiler_pass_pipeline.png}
	\caption{Dissertation figure used as a preliminary foundation: \enola{} compiler pass pipeline. The proposed work extends this pipeline for compartment-aware, interrupt-aware, data-flow-aware, and I/O-aware attestation.}

The preliminary systems demonstrate that the proposed research is technically feasible. The new intellectual contributions will come from integrating configurable hardware trust anchors with runtime behavior evidence, decomposing trusted software into attestable compartments, and validating the microarchitectural assumptions of modern embedded trust anchors.

\section{Intellectual Merit}
% ==========================================================
This project will advance embedded systems security in four ways. First, it will develop new abstractions and tools for synthesizing application-specific trusted execution environments from policies. Second, it will develop scalable runtime attestation algorithms that bind control-flow, data-flow, and I/O-flow behavior to a trusted execution substrate. Third, it will introduce compartmentalization as a first-class property of embedded trust anchors and provide evidence that compartment boundaries were enforced. Fourth, it will establish a systematic methodology for studying and defending against transient-execution leakage on modern MCUs. Together, these contributions will move the field from static trusted execution toward verifiable trusted behavior.

% ==========================================================
\section{Broader Impacts}
% ==========================================================
The proposed research will improve the security of embedded and cyber-physical systems used in healthcare, transportation, manufacturing, public infrastructure, and smart environments. The project will produce open-source tools, benchmarks, and educational materials when possible. The research will support graduate and undergraduate training in embedded security, hardware/software co-design, trustworthy computing, and microarchitectural security. The project will also contribute hands-on modules for courses and workshops on MCU security, FPGA-based trusted execution, and runtime attestation.
